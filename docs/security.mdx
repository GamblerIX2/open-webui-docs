Open WebUI 非常重视用户数据的安全性和机密性。我们的技术架构和开发流程旨在最大限度地减少漏洞，并维护相关方对我们的信任。定期的评估、代码库审查以及系统地采用最佳实践方法，有助于让安全性成为我们项目生命周期的核心部分。

我们结合了自动化和手动技术来跟进不断演变的威胁，并持续改进我们的方法，包括计划集成先进的静态分析和依赖分析工具。我们的重点是主动的风险管理以及对所报告的任何漏洞进行负责任的处理。

## 在何处及如何报告安全漏洞

:::info

Open WebUI 社区之所以能够繁荣，是因为有像你一样的人，你们深切关注如何让软件对每个人都更安全。

为确保你的发现能够真正帮助保护用户并得到迅速解决，请**仅通过**我们的 [官方 GitHub 安全页面](https://github.com/open-webui/open-webui/security) 提交所有安全漏洞报告。任何其他网站、服务或所谓的“赏金”平台都与我们**没有任何关联**，你在那里提交的重要成果根本无法传达给能解决问题的人。

我们知道，去信任那些做出各种承诺的平台很有诱惑力，但只有 GitHub 才能让你直接联系到守护 Open WebUI 的团队。为了让你的警觉真正造福整个社区，请在这里（真正重要的地方）报告问题。

:::

**Open WebUI 的所有安全漏洞必须专门通过我们的官方 GitHub 仓库进行报告：[https://github.com/open-webui/open-webui/security](https://github.com/open-webui/open-webui/security)。**

所有安全漏洞报告均通过我们官方的 GitHub 平台进行独家管理。在 GitHub 上集中处理披露信息，可以让项目维护者透明、高效、保密地处理每一份报告。这种方式为贡献者和用户提供了可见性、问责制，并确保与安全相关的沟通和解决方案都被全面记录和可靠管理。通过 GitHub 之外的任何平台、渠道或第三方服务提交的报告，都无法纳入我们的官方安全工作流，因此可能无法为 Open WebUI 的安全做出贡献。

### 为什么只用 GitHub？

我们采用单一的集中报告机制，源于技术严谨性和道德责任：

- **完整性与可追溯性：** 仅通过 GitHub 管理报告，可将每一个 Issue 都保留在开源生态系统中，社区可以在此验证其过程、结果和贡献者的问责制。
- **用户安全：** 其他声称促进漏洞披露、提供奖励或赏金计划的网站，与 Open WebUI 均无关联。向这类平台提交漏洞不仅无法使项目变得更安全，反而可能无意中将敏感细节置于被滥用或未经授权披露的风险之中。
- **保护社区：** 当信息绕过我们集中的安全工作流时，项目不仅更容易受到未修复漏洞的威胁，还会让误导性或剥削性行为得以蔓延。尽管外部网站声称充当中间人或支付赏金，但这些实体无法为用户提供任何保证，还可能打着奖励的幌子鼓励有问题或不安全的行为。通过我们的 GitHub 进行披露，是你的专业知识能够切实造福整个生态系统的唯一途径。

## 报告指南

**[你可以在此处找到最新版本的安全指南。](https://github.com/open-webui/open-webui/security)**

为了确保提交的是建设性、可操作且能切实提升安全性的报告，所有提交的内容必须满足以下要求：

1. **报告的内容必须是漏洞 (Vulnerability)**：安全漏洞是指一种可被利用的弱点，该弱点会导致系统以非预期的方式运行，从而允许攻击者绕过安全控制、获取未授权访问、执行任意代码或提升权限。配置选项、缺少的功能以及预期的协议行为不属于漏洞。

2. **拒绝含糊不清的报告**：类似于“我发现了一个漏洞”但没有任何细节的提交内容，将被视为垃圾信息并被拒收。

3. **要求深入了解**：报告必须反映出对代码库的清晰理解，并提供关于漏洞的具体细节，包括受影响的组件和潜在影响。

4. **必须提供概念验证 (PoC)**：每一次提交都必须包含一份记录良好的概念验证（Proof of Concept），以证明该漏洞的存在。PoC 必须说明跨越了哪条安全边界（机密性、完整性、可用性、真实性、不可否认性），该漏洞如何被滥用，以及攻击者现在可以执行什么操作。有效的 PoC 包括：带有确切命令的逐步复现说明、包含详细执行说明的完整利用代码，或演示漏洞利用的截图/视频（作为书面步骤的补充）。如果有机密性方面的担忧，鼓励报告者创建仓库的私有分支 (private fork) 并与维护者共享访问权限。缺乏有效证据的报告可能会被无视。如果我们很难用你的 PoC 复现该利用，我们会通知你；但如果我们多次尝试仍无法复现，该报告可能会被关闭。

5. **要求提供修复方案**：连同 PoC 一起，你必须提供 **补丁/PR** 或 **修复计划 (可操作的步骤)**，让维护者能够无需猜测地进行应用。你的修复建议可包含（例如）：可能的根本原因（错在哪里）、要修改的位置（如果知道的话，提供文件/模块/函数名称）、建议的修复方法（验证/清理规则、身份验证检查、安全默认值等），以及任何需要注意的安全权衡或潜在的回归问题。

6. **简化的合并流程**：当漏洞报告符合上述标准时，我们会像对待常规 Pull Request 一样，考虑将提供的补丁立即合并。结构良好且详尽的提交将加快我们提升安全性的进程。

7. **默认配置测试**：所有的漏洞报告都必须在 Open WebUI 的开箱即用（默认）配置下进行测试并可复现。那些仅在明确削弱安全设置的情况下才会出现的漏洞报告，可能会被忽略。不过，如果你认为自己发现的安全问题影响了默认配置、代表了对预期安全控制的真正绕过，或者只在非默认配置下生效但该配置很可能会在生产部署中使用，那么我们绝对希望能听到你的报告。这项政策是为了过滤掉配置错误和部署问题，而非阻止合法的安全研究。

8. **要求了解威胁模型**：报告必须展现出对 Open WebUI 自托管、需身份验证、基于角色的访问控制架构的理解。如果将 Open WebUI 与安全模型完全不同的服务进行比较，而不承认架构上的差异，可能会导致报告被拒绝。

9. **CVSS 评分的准确性**：如果你在报告中附带了 CVSS 评分，它必须准确反映该漏洞（根据 CVSS 方法论）。常见的错误包括：在需要身份验证时将其评为 PR:N (None)；对假设的攻击链而不是实际漏洞进行评分；或者在没有证据的情况下夸大严重性。我们会调整不准确的 CVSS 评分。故意夸大评分可能会导致报告被拒绝。如果你引用了其他 CVE 来支持你的报告，请确保它们在漏洞类型、威胁模型和攻击向量方面确实具有可比性。如果引用了不同产品类别、不同漏洞类别或不同部署模型的 CVE，会导致我们怀疑你的报告是 AI 生成的。

10. **管理员操作不在讨论范围内**：需要管理员主动执行不安全操作的漏洞，不被视为有效漏洞。管理员拥有完全的系统控制权，并且被期望了解其操作和配置的安全影响。这包括但不限于：添加恶意的外部服务器（模型、工具、Webhooks）；将不受信任的代码粘贴到 Functions/Tools 中；或故意削弱安全设置。那些需要管理员疏忽或对管理员进行社会工程学攻击的报告可能会被拒绝。然而，如果你认为你发现了一个影响管理员的漏洞，且该漏洞**不是**由管理员疏忽或故意恶意操作引起的，那么我们绝对希望了解它。这项政策旨在过滤针对管理员的社工攻击和类似恶意行为，而不是阻止合法的安全研究。

11. **AI 报告的透明度**：由于 AI 辅助漏洞报告的数量激增，如果**在任何环节**使用了 AI——无论是用于编写报告、生成 PoC 还是识别漏洞——你都**必须予以披露**。我们默认**不会拒绝**使用 AI 辅助的漏洞报告。但是，如果我们怀疑你使用了 AI 却没有向我们披露，我们会提出严厉的后续问题，以验证你对所报告的漏洞以及对 Open WebUI 本身的理解。如果我们怀疑你隐瞒了使用 AI 的情况，且你的报告最终被证明无效、不是漏洞或无法复现，你可能会被禁止在未来报告漏洞。采取这一措施是必要的，因为目前出现了大量明显由 AI 编写的漏洞报告，其中绝大多数都不是真正的漏洞，而是配置错误，且没有提供 PoC、违反了此处列出的规则、明显缺乏对 Open WebUI 的理解、在评论中提供相互矛盾的信息，或使用了不合逻辑的论点。

不合规的提交将被关闭，屡教不改的严重违规者可能会被封禁。我们的目标是营造一个建设性的报告环境，让高质量的提交能够提升所有用户的安全性。那些不仅能发现漏洞，还能提交稳健且随时可合并修复方案的贡献者，将有助于加快我们的响应速度并强化整个社区。

如果你觉得由于特定漏洞的原因，无法遵循所有列出的要求，也**请依然提交报告**——无论如何，我们都会检查每一份报告。

## 预期的响应时间范围

由于不断收到大量的漏洞报告、Issues、Discussions、Pull Requests 以及日常项目维护工作——加上最近大量无效的 AI 生成报告（参见上文的 AI 报告透明度）的叠加——我们的响应能力十分有限。Open WebUI 是一个由小团队维护的社区驱动项目，安全报告需要与所有其他项目职责并行处理。

**请预留几周的时间** 让我们对你的报告进行分类、调查、修复和发布。虽然我们的目标是尽快回应每一份报告，但**长达数周的沉默是正常的**。**这并不意味着你的报告被忽视了**——它仅仅意味着我们还没有抽出精力来处理它。从最初提交到最终发布，整个过程现实情况中可能需要好几周。我们非常感谢你的耐心与理解。

## 保密披露 (Confidential Disclosure)

通过 GitHub 安全公告 (Security Advisories) 提交的漏洞报告是**私密且机密的**。在公告**完全发布之前**（不仅仅是分配了 CVE ID 时，而是指该公告本身对公众可见时），**严禁公开披露**与已提交的漏洞报告相关的**任何**细节。

这项禁令适用于**所有渠道**，包括但不限于：

- 在 Pull Requests、Issues 或 Discussions（在 GitHub 或其他地方）发表评论
- 社交媒体、博客、论坛或任何其他网站
- Discord、Reddit 或任何其他平台、网站或服务

过早披露会损害所有 Open WebUI 用户的安全，并**破坏**负责任披露流程中固有的信任。**如果在官方发布之前公开披露漏洞细节，报告者<ins>可能会被永久禁止在未来提交报告。</ins>**

## 针对非漏洞相关的安全关切

如果你的疑虑不符合上述概述的漏洞要求，并非漏洞，但仍涉及安全方面的担忧，请使用以下渠道：

- **文档问题/改进想法**：在我们的 [文档仓库](https://github.com/open-webui/docs) 中提出 Issue
- **功能请求**：在 [GitHub Discussions - Ideas](https://github.com/open-webui/open-webui/discussions/) 中发起讨论，与社区探讨是否有很多人需要这个功能请求
- **配置求助**：在我们的 [Discord 服务器](https://discord.gg/5rJgQTnV4s) 或 [Reddit](https://www.reddit.com/r/OpenWebUI/) 上向社区寻求帮助与指导
- **一般性问题**：使用我们的 [Issue Tracker](https://github.com/open-webui/open-webui/issues)

不属于漏洞但仍与安全相关的关切的示例包括：建议更好的默认配置值、安全加固建议、部署最佳实践指导、不清晰的配置说明、需要额外的安全文档、关于可选安全增强功能（2FA、审计日志等）的功能请求，以及有关生产环境部署的一般安全问题。请为你遇到的具体问题使用正确的对应渠道。

## Tools, Functions 和 Pipelines 安全性

Open WebUI 通过 **Tools (工具)**、**Functions (函数，包括 Pipes, Filters, 和 Actions)** 以及 **Pipelines (管道)** 提供了强大的可扩展性。这些功能允许你使用自定义的 Python 代码来扩展 Open WebUI 的能力。然而，伴随这种强大能力而来的是安全责任。

:::warning

**Tools、Functions 和 Pipelines 会在你的服务器上执行任意 Python 代码。** 这是有意为之的——这也是它们之所以强大的原因。然而，这意味着它们拥有与 Open WebUI 后端进程本身相同级别的访问权限。

:::

### 安全隐患

当你安装 Tool、Function 或 Pipeline 时，你将赋予它以下能力：

- **访问文件系统** — 读取或写入后端进程能够访问的任何文件
- **发起网络请求** — 连接到外部服务，这可能存在数据外泄的风险
- **执行系统命令** — 通过子进程 (subprocess) 运行 Shell 命令
- **访问环境变量** — 读取 API keys、密钥和配置信息
- **修改数据库** — 访问或更改存储的数据
- **消耗计算资源** — 运行 CPU 密集型操作

### 最佳实践

| 实践 | 描述 |
|----------|-------------|
| **只从可信来源安装** | 仅使用来自官方社区库或你信任来源的 Tools/Functions |
| **安装前审查代码** | 在导入之前，阅读并理解 Tool/Function 的具体作用 |
| **限制工作区 (Workspace) 访问权限** | 仅向受信任的管理员授予 Workspace 权限 |
| **审计已安装的插件** | 定期审查已安装的 Tools (Workspace → Tools) 和 Functions (Admin Panel → Functions) |
| **保护你的数据目录** | `/app/backend/data` 目录包含了你的数据库和已缓存的插件——请保护它免遭未授权访问 |
| **监控资源使用情况** | 留意那些可能暗示加密货币挖矿或其他滥用行为的异常 CPU 峰值 |
| **使用官方 Docker 镜像** | 只从 `ghcr.io/open-webui/open-webui` 或 `openwebui/open-webui` (Docker Hub) 拉取镜像——非官方镜像可能被植入恶意代码 |

### 哪些不属于漏洞

以下情况**不被**认为是漏洞，因为它们需要管理员的主动操作（另请参阅上方报告指南中的 [管理员操作不在讨论范围内](#reporting-guidelines) 和 [要求了解威胁模型](#reporting-guidelines)）：

- 管理员安装了恶意的 Tool 或 Function
- 管理员将 Workspace 访问权限授予了不受信任的用户
- 管理员从不受信任的来源导入代码
- 具有对数据卷 (data volume) 写入权限的攻击者注入了恶意插件

更笼统地说，**任何描述涉及 Tools 或 Functions 的攻击链——包括但不限于代码执行、文件访问、网络请求或环境变量访问的报告，都将以“非漏洞 / 预期行为”为由被关闭。** 如果管理员将 Tool 权限授予了不受信任的用户，这构成了故意的错误配置。

这些场景代表的是**管理员的疏忽**或**环境被攻破**，而不是 Open WebUI 本身的漏洞。更多详细信息，请参阅 [我们的安全策略](https://github.com/open-webui/open-webui/security) 以及 [插件安全文档](/features/extensibility/plugin/)。

## 生产部署安全

Open WebUI 专为私有的受信任网络而设计，其默认设置经过优化以简化初始配置。处理敏感数据的生产部署，应审阅 [Hardening Guide (加固指南)](https://docs.openwebui.com/getting-started/advanced-topics/hardening) 中的综合检查清单。

一个关键的考量：**在没有配置 Redis 的情况下，注销账号和修改密码并不会撤销现有的 JWT token**——它们在过期前（默认：4 周）将一直有效。这意味着被泄露的 token 无法被主动失效，管理员发起的账号停用也不会立即切断访问。对于生产部署，请务必配置 [Redis](https://docs.openwebui.com/tutorials/integrations/redis) 或缩短 `JWT_EXPIRES_IN` 的时间，以限制风险暴露的窗口期。详情请参阅 [Token Revocation (令牌撤销)](https://docs.openwebui.com/getting-started/advanced-topics/hardening#token-revocation)。

## 产品安全流程

- 对我们的架构和 Pipeline 进行内部审查与定期的外部评估
- 对前端和后端进行自动化和手动测试
- 将主动的风险管理和代码审查纳入持续开发流程的一部分
- 持续改进并整合用于静态/动态分析的先进工具

如果你有需要立即采取行动的安全疑虑，请在我们的 [安全公告门户网站](https://github.com/open-webui/open-webui/security/advisories) 或 [Issue Tracker](https://github.com/open-webui/open-webui/issues) 中创建报告。
