# 新用户必读

你已经安装了 Open WebUI、连接好了提供商，也完成了第一次对话。接下来呢？

本页会带你了解那些真正决定体验差异的关键点：让 Open WebUI 从“一个聊天界面”变成“日常真正好用的一套系统”。这些内容都不是强制的——你完全可以继续直接聊天——但如果你曾经想过“Open WebUI 能不能做 X？”，答案几乎总是**可以，而且就靠下面这些能力**。

## 🧩 插件：扩展能力全貌 {#-plugins-the-extensibility-story}

Open WebUI 的核心设计刻意保持精简。你在演示里看到的很多“哇，这也行”的能力——自动翻译、token/成本追踪、图片生成按钮、自定义后处理、不属于 OpenAI 或 Ollama 的提供商集成——很多都**不是内置功能，而是插件**。对新用户来说，理解插件生态是最重要的一次认知升级。

插件主要分为两大家族，名字基本就说明了它的作用：

| 家族 | 做什么 | 运行位置 | 示例 |
| :--- | :--- | :--- | :--- |
| **Tools** | 给模型新的能力（“需要 X 时调用这个函数”） | 运行在 Open WebUI 进程内，由模型在对话过程中触发 | Langfuse / OpenLit 可观测性、Home Assistant 控制、arXiv / PubMed 检索、Wolfram Alpha、Jira / Linear 工单创建、针对你自己数据库的 SQL 查询 |
| **Functions — Pipes** | 在模型选择器里新增一个由自定义代码驱动的“模型” | 同上 | 模型路由 Pipe（根据提示词决定走便宜还是昂贵模型）、多步 Agent 循环、没有兼容 OpenAI 端点的企业内部 LLM 后端 |
| **Functions — Filters** | 在请求和/或响应经过时统一修改 | 同上，每一轮对话都会执行 | 上下文裁剪、PII 清洗、token / 成本统计、Langfuse tracing、响应重格式化 |
| **Functions — Actions** | 在每条消息下增加一个执行自定义代码的按钮 | 同上，在用户点击时执行 | “重新生成后续问题”“翻译回复”“固定消息”“保存到 Knowledge” |

**如何安装：** [**Open WebUI Community**](https://openwebui.com/) 提供了一键安装目录——选中一个项目，点击 “Get”，粘贴到 **Admin Panel → Functions**（或 **Tools**），启用后再配置它的 **valves**（插件设置）即可。

**什么时候该安装：** 当你开始想“要是 Open WebUI 能做 X 就好了”时，十有八九它已经能做——只是通过插件实现。**先去社区站点看看**：那里已经有数千个 Tools、Filters、Actions 和 Pipes，通常你要的功能已经有人写好了。即使没有完全匹配的，最接近的版本往往也只差十几二十行代码，你可以直接在管理面板里 fork 它。

## 🤖 任务模型：界面背后的隐形模型 {#-task-models-the-invisible-model-behind-the-ui}

这是安装完成后最值得优先调整的一项，因为默认配置会悄悄消耗你的金钱、延迟和耐心。

每当 Open WebUI 需要为界面功能做一小段“思考”——生成侧边栏里的聊天**标题**、生成**标签**、给出**后续问题建议**、驱动输入框中的**自动补全**——它都会调用一个 **Task Model**。默认情况下，这个任务模型就是你当前正在聊天的主模型，这意味着：

- 你昂贵的旗舰模型会在每次新建对话时，仅仅为了写出“购物清单”这样的标题而被调用。
- 如果你使用的是较慢的本地模型，每次按键都会感觉卡顿，因为自动补全在等一个 30B 参数模型回应。
- 推理模型（o1、r1、启用 extended thinking 的 Claude）可能会先花 5 秒“思考”，才吐出三词标题。

这些成本很容易被忽略，因为它们都发生在后台。优先把它们处理掉。

**修复方法：** 在 **Admin Panel → Settings → Interface** 中设置专门的 Task Model。这里有两个字段，因为最佳选择取决于你的主聊天模型类型：

- **Task Model (External)** —— 当你使用云端模型（OpenAI、Anthropic 等）聊天时使用。建议设置为快速、便宜、**非推理型**的云模型，比如 `gpt-5-nano`、`gemini-2.5-flash-lite` 或 `llama-3.1-8b-instant`。
- **Task Model (Local)** —— 当你使用本地模型（Ollama、llama.cpp、vLLM）聊天时使用。建议设置为体积很小的本地模型，比如 `qwen3:1b`、`gemma3:1b` 或 `llama3.2:3b`。

主聊天体验不会改变，只是后台这些“杂活”不再拖慢你。

既然已经进入 Interface 设置，如果你的机器配置较低，或者你根本不需要其中某些功能，也可以直接把这些后台任务整体关掉。每项功能同时提供**管理面板开关**和**环境变量**：

| 后台任务 | 管理面板开关（Settings → Interface） | 环境变量 |
| :--- | :--- | :--- |
| 自动补全 *(最大头——每次按键都会触发)* | **Autocomplete Generation** | `ENABLE_AUTOCOMPLETE_GENERATION=False` |
| 后续问题建议 | **Follow-up Generation** | `ENABLE_FOLLOW_UP_GENERATION=False` |
| 聊天标题生成 | **Title Generation** | `ENABLE_TITLE_GENERATION=False` |
| 标签生成 | **Tags Generation** | `ENABLE_TAGS_GENERATION=False` |

在硬件较弱的环境中，自动补全是最值得优先关闭的“提速开关”——它每次按键都会触发，所以慢速任务模型会让整个输入框像陷进糖浆一样迟钝。

更多说明：[性能与内存 → 专用 Task Models](/troubleshooting/performance#1-dedicated-task-models)。

## 🧠 上下文管理：为什么长对话会崩 {#-context-management-why-long-chats-break}

对话来回足够多之后，你迟早会看到这样的报错：

> `The prompt is too long: 207601, model maximum context length: 202751`

这个错误**不是** Open WebUI 抛出的——它来自你的模型提供商。每次发送新消息时，*整个*会话（system prompt + 所有历史轮次 + 附件文件 + 工具调用结果 + 你的新消息）都会一起作为 “prompt” 发给模型。当总量超过模型的上下文窗口时，提供商就会拒绝请求。

Open WebUI 有意不内置统一裁剪器，因为：

- 每个模型的 **tokenizer 都不同**（GPT ≠ Claude ≠ Gemini ≠ GLM ≠ Llama）。
- 每个模型的 **上下文窗口也不同**（8k → 1M）。
- 每个部署对裁剪策略的需求都不同（按 token 裁、按轮次裁、按消息数裁、优先丢附件、总结旧消息等）。

并不存在一个“唯一正确”的答案。官方支持的方式是安装一个**过滤器 Function**，按照你的规则裁剪会话。社区里已经有不少覆盖常见策略的过滤器可以一键安装；如果都不完全合适，代码通常也足够短，拷贝后改一下就能用。

➡️ 完整指南（含可直接粘贴到实例中的最小版“保留最新 N 轮”过滤器）：[故障排查 → 上下文窗口 / Prompt Too Long](/troubleshooting/context-window)。

## 🔧 工具调用：让模型开始做事 {#-tool-calling-letting-the-model-do-things}

工具调用会把 LLM 从“一个非常聪明的文本框”变成“一个真的能查资料、跑代码、执行动作的助手”。你把 **Tool** 挂到对话或模型上，模型会在回复过程中自行决定何时调用、传什么参数。Open WebUI 执行它，把结果返回，再由模型继续生成。

每个新用户都应该优先完成下面两件事：

### 1. 打开 Native tool calling

Open WebUI 界面中有两种工具调用模式：**Native** 和 **Default**。**Default 已是遗留模式，不再受支持**——它还留在下拉菜单里，只是为了让旧部署在迁移期间不至于立即中断。**所有模型都应该配置为使用 Native Mode。** Native 更快、可保留 KV cache、支持内置系统工具（Memory、Notes、Knowledge、Web Search、Image Gen、Code Interpreter），也是未来唯一持续获得新特性的模式。

主流模型基本都支持它——OpenAI、Anthropic、Gemini、Llama 3.1+、Qwen 2.5+、DeepSeek、GLM，以及几乎所有当前主流模型。开启方式如下：

- **最佳方式——一次设置，应用到所有模型：** 在 **Admin Panel → Settings → Models** 中，点击模型列表**右上角**的 **Settings** 按钮。这里打开的是**全局模型参数**——把 **Function Calling = Native** 设好并保存后，实例中当前和未来新增的所有模型都会继承这个设置，无需逐个点进去改。
- 单模型覆盖：**Admin Panel → Settings → Models → [你的模型] → Model Parameters → Function Calling = Native**
- 单对话覆盖：在聊天页右侧边栏的 **Chat Controls** 中设置

如果一个本来有能力调用工具的模型“怎么都不调用”，90% 的情况下只是 Native Mode 没打开。如果某个特定的小模型在 Native Mode 下表现吃力，正确的解决办法是换一个更强的模型来做工具型对话——而不是退回 Default Mode。

### 2. 安装几个实用 Tools

很多大家第一时间想要的能力——**网页搜索**、**代码执行**、**图片生成**、**记忆**、**知识库检索**——其实已经是内置功能，不需要社区插件。你只需要在 **Admin Panel → Settings** 中启用它们（Web Search、Code Interpreter、Images 等），再把它们挂到模型上。开启 Native Mode 后，它们会自动作为内置系统工具注入。

[社区站点](https://openwebui.com/) 上的 Tools，则是用来补足那些*不是内置*的能力。值得优先探索的方向包括：

- **可观测性 / 成本追踪** —— Langfuse、OpenLit、Portkey 这类工具可以把每轮对话、token 使用量和延迟记录到你自己的观测栈里。一旦实例开始被多人使用，它们就非常关键。
- **智能家居 / 自动化集成** —— Home Assistant 工具可以让模型在对话中真正控制设备、例程和场景。
- **研究检索** —— arXiv、PubMed、Semantic Scholar、Wolfram Alpha——模型获得的是结构化结果，而不是训练数据里的模糊记忆，还带真实引用。
- **Issue / 工单 / 消息系统集成** —— 创建 Jira / Linear / GitHub issue、发 Slack 或 Discord、发送邮件——模型不再只是只读助手。
- **数据库 / API 工具** —— 给模型暴露一个只读 SQL 查询工具，或一个访问你内部 API 的工具——这样它就能基于你的真实数据回答问题。
- **领域型工具** —— 天气、股票、时间、加密货币价格、物流追踪、菜谱 API，任何贴合你工作场景的能力。

Tools 会出现在聊天输入框旁边的 `+` 菜单中。只启用当前对话需要的工具；模型只能看到你已启用的那些工具。

:::tip 认真一点——去逛逛社区站点
上面列的只是一小部分。[**Open WebUI Community**](https://openwebui.com/) 上有**数千个**社区构建的 Tools、Filters、Actions 和 Pipes，覆盖的使用场景远超核心团队能想到的范围。真正要自己写东西之前，**先去社区里搜一下**——按热度排序、按分类筛选、翻几页看看。通常你要的功能已经存在，或者只差两行。点一下安装，调好 valves，就结束了。

这也是为什么要把 Open WebUI 看成一个平台，而不只是一个应用。真正的功能集，就是社区本身。
:::

## 📚 基础 RAG：与自己的文档对话 {#-basic-rag-chatting-with-your-own-documents}

**RAG**（Retrieval-Augmented Generation，检索增强生成）让你可以说出“这是一个 400 页的 PDF，请回答我关于它的问题”，而不需要模型每轮都把整份文档读一遍（它也做不到）。Open WebUI 会把文档切块、生成向量嵌入、存入向量数据库，并在对话时只检索相关片段再传给模型。

最简单的两种使用方式如下：

1. **一次性附件。** 把文件拖进任意聊天输入框，然后提问。文件会只为这场对话完成切块和嵌入。
2. **知识库。** 对于需要跨多个聊天反复复用的文档（公司手册、代码库、研究资料、用户手册），进入 **Workspace → Knowledge** 创建知识库。之后你可以在聊天里通过输入框中的 `#` 快捷方式把**整个知识库**附加进来，或者在 **Workspace → Models** 中把它绑定到某个模型上，让该模型始终可用。

默认配置足够你顺利上手。等你开始遇到瓶颈后，最重要的是下面三个调节点：

- **Embedding 引擎。** 默认使用的 SentenceTransformers `all-MiniLM-L6-v2` 会在本地 CPU 上运行，每个 worker 大约消耗 500 MB 内存。只要进入多人部署，就建议通过 `RAG_EMBEDDING_ENGINE` 改为外部 embeddings API（OpenAI，或使用 `nomic-embed-text` 的 Ollama）。
- **内容提取引擎。** 默认使用 `pypdf`，在高强度导入时会内存泄漏。只要不再是轻量尝试性质的使用，就建议通过 `CONTENT_EXTRACTION_ENGINE` 切换到 **Tika** 或 **Docling**。
- **向量数据库。** 默认的 ChromaDB（本地 SQLite 支撑）无法可靠承受多 worker 部署。要扩展时，请使用 **Milvus**、**Qdrant** 或 **PGVector**。

对于“一个用户、几份 PDF”这种场景，这些都不重要；但一旦你有 100 份文档，或者 10 个并发用户，它们就会迅速变得重要。

## ⚡ Open Terminal：给模型一台真正的电脑 {#-open-terminal-giving-the-model-a-real-computer}

如果“运行 Python”对你来说太受限，你希望模型真的能在机器上工作——clone 仓库、安装包、跑测试、打开本地网页预览、对真实 CSV 做数据分析——那就该看看 [**Open Terminal**](/features/open-terminal)。它会把一个真正的 shell（默认运行在 Docker 容器沙箱里，也可改为裸机）作为工具暴露给模型，就像它调用其他工具一样来使用。还附带聊天内文件浏览器、实时网页预览和技能定义。

这是很多用户从基础聊天进一步深入后最容易产生“原来还能这样”的功能。它把 Open WebUI 从聊天界面变成了模型真正帮你做事的地方。只要 Native Mode 已开启，且你给了模型一个足够能干的终端，就可以直接让它为你搭一个小应用，或者对一整个文件夹跑分析，然后看它一步步完成。

## 接下来做什么

第一天并不需要把上面所有内容都配齐。一个比较合理的新安装顺序是：

1. **第一天：** 先选一个不错的默认模型，聊上几轮，熟悉界面。
2. **紧接着：** 设置 Task Model，并决定哪些后台任务真的需要开启。这是最明显的“体感更顺手”优化，也能直接减少每次对话的隐藏成本。
3. **第一周内：** 全局开启 Native Mode，再装一两个和你工作场景匹配的 Tools。
4. **第一次碰到时：** 一旦看到 “prompt is too long”，就去装一个上下文过滤器。
5. **需要时：** 第一次想跨多份文档提问时，就搭一个 Knowledge 知识库。
6. **准备玩大一点时：** 把模型接到 Open Terminal，让它真的帮你干活。
7. **开始扩展后：** 如果不再是单用户使用，就回头重新看一遍 RAG 基础设施部分。

其他内容——企业级 SSO、多副本高可用、Redis 扩展、可观测性——都在 [高级主题](/getting-started/advanced-topics) 和 [故障排查](/troubleshooting) 里，等你需要的时候再深入即可。
